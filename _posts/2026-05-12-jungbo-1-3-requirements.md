## 1. 요구사항의 종류

| 구분 | 설명 | 예시 |
|------|------|------|
| 기능적 요구사항 | 시스템이 무엇을 해야 하는가 | 로그인, 결제, 검색 기능 |
| 비기능적 요구사항 | 시스템이 어떻게 동작해야 하는가 | 성능, 보안, 가용성, 유지보수성 |

---

## 2. 요구사항 개발 프로세스

**도출(Elicitation) → 분석(Analysis) → 명세(Specification) → 확인(Validation)**

### 도출 기법
- 인터뷰, 설문조사
- 브레인스토밍, 델파이 기법
- 프로토타이핑
- 유스케이스

### 분석 기법
- **구조적 분석**: DFD(데이터 흐름도), 데이터 사전 사용
- **객체지향 분석**: 유스케이스 다이어그램, 클래스 다이어그램 사용

---

## 3. 요구사항 명세 (SRS)

SRS(Software Requirements Specification): 요구사항 명세서

**좋은 요구사항의 특성**

| 특성 | 설명 |
|------|------|
| 완전성 | 필요한 모든 요구사항 포함 |
| 일관성 | 요구사항 간 모순 없음 |
| 명확성 | 이해 가능하고 모호하지 않음 |
| 검증 가능성 | 테스트로 충족 여부 확인 가능 |
| 추적 가능성 | 출처와 관련 구현 요소 추적 가능 |

---

## 4. 요구사항 분석 도구

### DFD (Data Flow Diagram, 데이터 흐름도)
- 시스템 내 데이터의 흐름을 도식화
- 구성 요소: 프로세스(원), 데이터 저장소(평행선), 외부 엔티티(사각형), 데이터 흐름(화살표)

### 데이터 사전 (Data Dictionary)
- DFD에 등장하는 데이터 요소를 정의한 목록
- 기호: `=`(정의), `+`(구성), `[ ]`(선택), `{ }`(반복), `( )`(생략 가능)

---

## 5. 요구사항 확인 및 검증 기법

| 기법 | 설명 |
|------|------|
| 요구사항 검토 (Review) | 동료 검토, 워크스루, 인스펙션 |
| 프로토타이핑 | 시제품으로 요구사항 확인 |
| 테스트 케이스 | 요구사항 충족 여부 테스트 설계 |
| CASE 도구 | 자동화 도구로 일관성·완전성 검토 |

### 검토 방법 비교

| 방법 | 설명 |
|------|------|
| 동료 검토 (Peer Review) | 작성자가 직접 설명, 동료가 검토 |
| 워크스루 (Walkthrough) | 사전 배포 후 회의에서 검토, 비공식적 |
| 인스펙션 (Inspection) | 훈련된 검토자가 체계적으로 검토, 가장 공식적 |

---

## 6. 요구공학 (Requirements Engineering)

요구사항을 정의, 문서화, 유지하는 프로세스 전체를 의미.

- **요구사항 추적 매트릭스 (RTM)**: 요구사항과 구현/테스트 항목 간 연결 관계를 표로 관리
- **요구사항 변경 관리**: 변경 요청 → 영향 분석 → 승인 → 반영 절차 준수
