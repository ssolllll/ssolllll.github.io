## 1. 디자인 패턴 개요

GoF(Gang of Four)가 제시한 객체지향 설계에서 자주 발생하는 문제의 재사용 가능한 해결책 23가지.

**분류**: 생성(Creational) / 구조(Structural) / 행위(Behavioral)

---

## 2. 생성 패턴 (Creational Pattern)

객체 생성 방식을 캡슐화하여 유연성 제공.

| 패턴 | 설명 |
|------|------|
| **Singleton** | 클래스 인스턴스를 오직 하나만 생성, 전역 접근점 제공 |
| **Factory Method** | 객체 생성을 서브클래스에 위임, 인터페이스로 생성 |
| **Abstract Factory** | 관련 객체 집합을 생성하는 인터페이스 제공 |
| **Builder** | 복잡한 객체를 단계별로 생성, 동일 프로세스로 다양한 표현 |
| **Prototype** | 기존 객체를 복사(Clone)하여 새 객체 생성 |

---

## 3. 구조 패턴 (Structural Pattern)

클래스나 객체를 조합하여 더 큰 구조를 만드는 패턴.

| 패턴 | 설명 |
|------|------|
| **Adapter** | 호환되지 않는 인터페이스를 연결 (변환기 역할) |
| **Bridge** | 구현과 추상을 분리하여 독립적으로 변화 |
| **Composite** | 개별 객체와 복합 객체를 동일하게 처리 (트리 구조) |
| **Decorator** | 객체에 동적으로 기능 추가 (상속 대안) |
| **Facade** | 복잡한 서브시스템에 단순한 인터페이스 제공 |
| **Flyweight** | 공유를 통해 다수의 세밀한 객체를 효율적으로 지원 |
| **Proxy** | 다른 객체에 대한 대리인(접근 제어, 지연 초기화 등) |

---

## 4. 행위 패턴 (Behavioral Pattern)

객체 간 알고리즘이나 책임 분배를 정의하는 패턴.

| 패턴 | 설명 |
|------|------|
| **Chain of Responsibility** | 요청을 처리할 객체를 체인으로 연결, 순서대로 처리 |
| **Command** | 요청을 객체로 캡슐화, 실행 취소(Undo) 가능 |
| **Interpreter** | 언어 문법을 클래스로 표현하여 해석 |
| **Iterator** | 내부 구조 노출 없이 컬렉션 요소 순회 |
| **Mediator** | 객체 간 직접 통신 대신 중재자를 통해 통신 |
| **Memento** | 객체 상태를 저장·복원 (Undo 기능) |
| **Observer** | 상태 변화 시 의존 객체들에게 자동 알림 (pub-sub) |
| **State** | 상태에 따라 객체 행동 변경 |
| **Strategy** | 알고리즘을 캡슐화하여 상호 교환 가능 |
| **Template Method** | 알고리즘 골격 정의, 일부 단계를 서브클래스에 위임 |
| **Visitor** | 객체 구조를 변경하지 않고 새 연산 추가 |

---

## 5. 자주 출제되는 패턴 핵심 정리

| 패턴 | 핵심 키워드 |
|------|------------|
| Singleton | 하나의 인스턴스, 전역 접근 |
| Factory Method | 객체 생성을 서브클래스에 위임 |
| Observer | 상태 변화 → 자동 알림, 1:N 관계 |
| Strategy | 알고리즘 교체 가능, 캡슐화 |
| Decorator | 동적으로 기능 추가, 상속 대안 |
| Facade | 복잡한 시스템을 단순 인터페이스로 |
| Proxy | 대리인, 접근 제어 |
| Adapter | 인터페이스 변환, 호환성 |
| Template Method | 골격 정의, 일부 단계 위임 |
| Command | 요청 객체화, Undo 지원 |

---

## 6. 디자인 패턴 vs 아키텍처 패턴

| 구분 | 디자인 패턴 | 아키텍처 패턴 |
|------|-------------|---------------|
| 범위 | 클래스/객체 수준 | 시스템 전체 수준 |
| 예시 | GoF 23가지 | MVC, 레이어드, MSA |
